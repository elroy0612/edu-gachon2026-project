# 시스템 아키텍처 설계 노트

전체 아키텍처 개요는 README 7~9절 참고. 이 문서는 **왜 이렇게 설계했는지에 대한 결정 사항**을 기록합니다.

---

## Tool Calling(ReAct 동적 루프)을 쓰지 않는 이유

### 배경

`app/tools/schemas.py`에 `ToolCall`(LLM이 호출할 도구 이름·입력을 구조화 출력으로 정의하는 스키마)이
만들어져 있고, `app/agents/react_loop.py`도 이름은 "ReAct Loop"입니다. 하지만 실제로는 **둘 다
LLM이 아니라 개발자가 코드에 순서를 하드코딩**해서 실행합니다 (`run_tool("search_places", ...)`처럼
문자열을 직접 호출). 즉 `ToolCall` 스키마는 정의만 되어 있고 실제로는 어디서도 쓰이지 않는 상태입니다.

### 결정: Tool Calling 방식을 도입하지 않는다

TripRoute의 여행 일정 생성 절차는 **요청마다 절차가 달라지지 않습니다.** 항상 다음 순서로 고정됩니다:

```
입력 분석(Coordinator) → 취향 검색(RAG) → 동선 계산(Route Planner) → 비용 계산(Financial) → 최종조립(Coordinator)
```

Tool Calling(ReAct 패턴)은 **매 요청마다 필요한 절차 자체가 달라지는 범용 에이전트**(예: 어떤 질문엔
계산기가 필요하고 어떤 질문엔 검색이 필요한 챗봇)에 적합한 방식입니다. 우리처럼 절차가 항상 고정된
경우 LLM에게 "다음에 뭘 할지" 판단까지 맡기면:

- **속도 저하**: 매 단계마다 "다음에 뭘 부를지" 판단하는 LLM 호출이 추가로 필요함
- **비용 증가**: 그 판단용 호출 자체가 chat API 비용(임베딩보다 훨씬 비쌈)
- **예측 불가능성**: 가끔 LLM이 순서를 잘못 판단하거나 불필요한 도구를 호출할 위험

이런 단점만 생기고 얻는 이점이 없습니다. 그래서 **LangGraph의 고정된 그래프(node/edge)로 흐름을
설계 시점에 확정**하고, 각 Agent 내부에서는 LLM(Upstage)을 "다음 행동 결정"이 아니라 **좁은 범위의
텍스트 작업**(자연어 조건 추출, 추천 이유 문장 생성, 비정형 요금 텍스트 구조화)에만 씁니다.

### 두 패턴 비교

| | ReAct / Tool-calling 방식 | TripRoute의 오케스트레이션 방식 |
|---|---|---|
| 제어 흐름 결정 주체 | LLM이 매번 실시간으로 결정 | Coordinator + LangGraph edges가 설계 시점에 고정 |
| 적합한 경우 | 매번 필요한 절차가 달라지는 범용 업무 | 절차가 항상 동일한 정형화된 업무 (우리 경우) |
| LLM의 역할 | "다음 행동"까지 결정 | 정해진 각 단계 안에서 좁은 텍스트 작업만 수행 |

### `app/tools/` 폴더는 어떻게 하나

`ToolCall`은 실제로 안 쓰이지만, 지금 당장 지우지 않고 유지합니다. `mock_tools.py`/`schemas.py`는
Step 2에서 실제 서비스(`tour_api.py` 등)로 교체되는 과정에서 자연스럽게 정리될 예정이므로,
그 시점에 팀 논의를 거쳐 삭제 여부를 다시 결정합니다. (관련: `project_plan.md`의
"Mock ReAct Loop 프로토타입" 섹션 할 일 목록)
